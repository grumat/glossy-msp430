# DtrigSbw — SPI-stream alternate (design note)

**Status:** alternate design / future exploration. Not the primary path;
that is the BSRR-script approach in [DTRIG_SBW_DRIVER.md](DTRIG_SBW_DRIVER.md).

This note captures an alternate SBW encoding that reuses almost the entire
`DtrigJtag` machinery instead of building a parallel BSRR-DMA framework. It
is **only viable on buffered boards** (Bluepill, BlackPill-BMP, Nucleo-G431,
Nucleo-L432) where the PCB already routes SBWDIO through the SPI MOSI/MISO
pins plus the SBWO mux GPIO. STLinkV2's single bidirectional pin cannot use
this path.

## Premise

The buffered hardware has:

- **SBWDIO_OUT** on SPI1_MOSI (PA7 on F1 boards)
- **SBWDIO_IN**  on SPI1_MISO (PA6 on F1 boards)
- **SBWCLK**    on SPI1_SCK   (PA5 on F1 boards)
- **SBWO**      on a TIM1 channel-capable GPIO

`SBWO` selects which physical pin reaches the external SBWDIO trace:
high = output (PA7 drives), low = input (PA6 samples).

**SBWO must be timer-capable** — the whole point of this design is that the
direction flip is generated by a TIM channel (phase-locked to the SPI bit
clock), not GPIO bit-bang. Verified pin mapping:

| Board (target) | SBWO pin | Timer channel |
|----------------|----------|---------------|
| BluePill-G431 (`target.bluepill.g431kb`) | PA10 = `SBW_RD` | **TIM1_CH3** (AF6); also TIM2_CH4 (AF10) |
| Bluepill F103 (`target.bluepill`) | PA10* | **TIM1_CH3** |
| Nucleo-L432 (retired) | PA10 | TIM1_CH3 (AF1) |

\* the current `target.bluepill` still maps SBWO to PA9 (old layout) — it
needs the remap to match the BluePill-G431 board where SBW_RD = PA10.

PA10 = TIM1_CH3 is a **regular channel** (not a complementary CHN), so the
SBWO PWM uses the same non-complementary path as `kWaveJtagTmsCmpComplementary
= false` on these boards. No resource clash with JTAG: the g431 `platform.h`
already uses TIM1_CH3 in *JTAG* mode as a compare-only DMA trigger
(`Output::kDisabled`, PA10 left GPIO); JTAG and SBW are mutually exclusive, so
in SBW mode CH3 is free to drive PA10 directly.

The DtrigJtag design already drives JTCK + JTDI + JTDO via SPI1 with a TIM1
channel running phase-locked to toggle TMS at the right bit boundaries. The
core insight here: **SBW is structurally identical to JTAG if you let SPI
clock at 3× the SBW bit rate and use the TIM1 channel to toggle SBWO once
per SBW bit instead of TMS.**

## Frame mapping

One SBW bit = 3 SBWCLK cycles = 3 SPI bits at 3× clock. Per SBW bit:

| Phase | SPI cycle | MOSI value     | MISO sample | SBWO state |
|-------|-----------|----------------|-------------|------------|
| TMS   | 3k+0      | TMS_k          | (ignored)   | high (OUT) |
| TDI   | 3k+1      | TDI_k (= TCLK in DR / IR fill)  | (ignored) | high (OUT) |
| TDO   | 3k+2      | X (don't care) | TDO_k       | **low (IN)** |

So the SPI MOSI stream is the same pack-and-`__REV` encoding as DtrigJtag
but at 3 bits per logical JTAG bit instead of 1. The SPI MISO stream
captures TDO at every 3rd bit and the encoder discards the other 2/3 in
`GetResult()` (same masking pattern as DtrigJtag's payload window).

SBWO is generated by a TIM1 channel running at the SBW bit rate (1/3 of
the SPI rate), in PWM mode, low for 1/3 of the period (the TDO phase).
Phase-aligned to the SPI bit clock via the same TIM/SPI activation order
used for TMS in DtrigJtag.

## Clock-rate budget

> **Corrected ceiling.** An earlier draft quoted "5 MHz SBW ceiling." That was
> wrong: the MSP430 silicon limit is **SBWTCK ≤ 20 MHz**. Because SBW packs
> 3 SBWTCK cycles per logical JTAG-TCK bit, the logical wire rate ceiling is
> **SBWTCK/3 ≈ 6.6 MHz**, which also stays under the device's ~10 MHz JTAG
> TCK max. So the hard wall is ~6 MHz logical, not 5.

The real-world limiter is usually **not** SBWTCK and **not** the host SPI
divisor — it is whatever loads the SBWTDIO (= target RST/NMI) line. See
"Real limiter: SBWTDIO loading" below. Assuming a properly decoupled target
(series-isolated reset cap), the budget is:

- SPI bit clock = SBWCLK (one SBW sub-phase per SPI bit)
- Logical SBW bit rate = SBWCLK / 3

**F103** (APB2 = 72 MHz), SPI1 baud = PCLK/2ⁿ → usable SBWCLK grades:

| SPI baud | SBWCLK | Logical SBW bit | TDO-slot low-time | Notes |
|----------|--------|-----------------|-------------------|-------|
| PCLK/8   | 9 MHz  | 3 MHz           | ~56 ns            | **safe validated baseline** |
| PCLK/4   | 18 MHz | 6 MHz           | ~28 ns            | reach grade; near 20 MHz wall, needs `t(SBW,Rtd)` + LA check |
| PCLK/2   | 36 MHz | 12 MHz          | —                 | over silicon ceiling — out |

15 MHz is not on the F103 divisor grid; aim 9 MHz baseline / 18 MHz reach.

**G431** (APB2 up to 85 MHz) and **L432** (80 MHz) offer finer prescaler/
divisor choice and headroom up to the 20 MHz SBWTCK wall — the better hosts
for chasing the 6 MHz-logical limit. Speed grades drop in lockstep with
DtrigJtag's existing ones — the same per-grade `Init_N`/`ApplySpeed()`
machinery applies.

## Real limiter: SBWTDIO loading

SBWTDIO is the target **RST/NMI** pin, and most MSP430 application boards put
a reset filter on it. The filter topology decides whether fast SBW is even
possible:

- **Cap directly on the line** (e.g. 47 kΩ↑VCC ‖ 2.2 nF↓GND on the RST node):
  the driver must slew 2.2 nF every edge. Through a plausible 100–330 Ω
  series/damping path that is τ ≈ 0.2–0.7 µs, ~90 % settle in ≈ 0.5–1.6 µs →
  SBW bit rate stuck at a **few hundred kHz**. Passive pull-up recovery is
  hopeless (47 kΩ·2.2 nF = 103 µs), so both edges must be actively driven.

- **Series-isolated cap** — the TI-recommended SBW network (SLAU320): a series
  resistor (our target boards use **3k3**) sits between the RST/SBWTDIO node
  and the 47 kΩ/2.2 nF filter. The 3k3 + 2.2 nF low-pass corner is
  `1/(2π·3k3·2.2nF) ≈ 21.9 kHz`, so above ~22 kHz the **cap is decoupled**
  from the signal node. The cap node drifts to the line's mean voltage
  (τ = 3k3·2.2 nF ≈ 7.3 µs) and just sits there; the driven node only sees
  the 3k3 as a weak pull toward mid-rail (sub-mA, trivially overcome) plus the
  real ~tens of pF of pin/trace/translator. Edges are then **driver-limited**
  (τ ≈ 3–17 ns for 100–330 Ω into 30–50 pF), not RC-limited.

**Our target boards carry the 3k3-isolated network, so SBW is driver-limited,
not cap-limited** — which is why pushing for higher speed is realistic here.
On a target that wires the cap directly to RST, no probe architecture can
beat the few-hundred-kHz wall; that is a property of the target, not the probe.

With the cap decoupled, the binding constraints become:

1. **SBWTCK ≤ 20 MHz** silicon (→ ~6 MHz logical wall).
2. **`t(SBW,Rtd)`** — the MSP430's TDO-valid delay after the SBWTCK edge in the
   read slot — vs the SBWCLK low-time (28 ns @ 18 MHz, 56 ns @ 9 MHz). **Look
   this up per target family before committing to 18 MHz.** Uniform SPI cannot
   stretch only the TDO slot, so if `t(SBW,Rtd)` exceeds the uniform low-time
   the whole clock must drop — or switch to the TIM-clocked asymmetric variant
   (below).
3. **Turnaround/contention window** — host output-buffer disable must complete
   before the target drives; needs LA validation (see open question #1).

## Reused DtrigJtag infrastructure

If this approach is taken, the SBW driver shrinks dramatically:

| DtrigJtag component                | SBW reuse                                  |
|------------------------------------|--------------------------------------------|
| `SpiJtagDevType<>`                 | Reused verbatim, just at 3× the wire rate  |
| TIM1 CH PWM for TMS (CH3/PA10)     | Repurposed for SBWO on the same CH3/PA10 (period = 3 SPI bits) |
| TIM1 CHx DMA reload for TMS shape  | Repurposed for SBWO shape (single low pulse per period) |
| SPI RX DMA for JTDO                | Reused — captures every SBW cycle, software discards 2/3 |
| SPI TX DMA for JTDI                | Reused — emits the 3× pack-and-`__REV` stream |
| `JtclkWaveGen` flash-burst         | Reused as-is for SBW flash TCLK            |
| `JtagPending<T>` / `JtagWaitTransfer` | Reused as-is                            |
| `PingPongBuffer<uint8_t, ...>`     | Reused (back to byte-oriented like JTAG)   |

What goes away:

- The dedicated BSRR script DMA channel (Issue #2 primary design)
- The direction-script DMA channel
- The IDR sample DMA channel
- The `DirPolicy` abstraction (replaced by a fixed TIM channel PWM shape)
- The per-frame `uint32_t` buffer (back to `uint8_t` like JTAG)

## Cost / risk

**Advantages**
- ~3 fewer DMA channels consumed.
- The encoder is a small modification of DtrigJtag, not a new template.
- Phase alignment is solved (same trick as TMS) — no novel timing analysis.
- Pack-and-`__REV` benchmarks already validated on F103 (see
  [`feedback_pack_and_rev_encoding`](../../../C:/Users/Mathias/.claude/projects/P--github-com-glossy-msp430/memory/feedback_pack_and_rev_encoding.md)
  in user memory).

**Disadvantages / open questions**
- **Buffered hardware only.** STLinkV2 still needs the BSRR-or-bit-band
  path. Two codepaths to maintain unless STLinkV2 is dropped from the SBW
  roadmap (current state: not planned for SBW).
- **TDO read slot is the binding speed gate (uniform clock).** A fixed SPI baud
  forces all three sub-phases to the same low-time; the target-driven TDO slot
  needs ≥ `t(SBW,Rtd)` to present valid data (28 ns @ 18 MHz, 56 ns @ 9 MHz
  SBWCLK). If the datasheet delay exceeds the uniform low-time, the *entire*
  clock must drop — write/shift-in traffic gets penalised for the read slot.
  This is the main reason to keep the TIM-clocked asymmetric variant (below)
  as a contingency. On 3k3-isolated targets the cap no longer adds to this, so
  the uniform stream likely survives at the 9 MHz baseline; validate before 18.
- **SBWO PWM phase margin** vs MSP430's half-duplex turnaround window. The
  buffer enable has to transition during the gap *between* the TDI SPI bit
  and the TDO sample — needs LA validation that the SPI bit boundary and
  the TIM1 CHx falling edge land in the safe slot. DtrigJtag's TMS edge-
  alignment work is the analogue; the same `cnt_offset` calibration knob
  per speed grade applies.
- **TDI-phase MOSI must hold high for the head/tail "fill" bits.** Same as
  DtrigJtag — handled by `OnSet/ClearTclk` latching the fill value into the
  encoder.
- **SBW protocol expects a deterministic SBWCLK low-time during the TDO
  phase for the target to drive.** SPI CPOL/CPHA settings need to be picked
  so SBWCLK has the right shape during all three sub-phases, not just the
  TDI phase. May require a different SPI mode than DtrigJtag's mode 3.

## Decision

For the v1 SBW bring-up we are going with the **BSRR-script** approach
(see [DTRIG_SBW_DRIVER.md](DTRIG_SBW_DRIVER.md)) because:

1. It works on every board including STLinkV2 (via bit-banding).
2. The DMA-channel budget is not currently a constraint on F103.
3. Phase alignment is decoupled from SPI bit shape — fewer unknowns at
   bring-up.

This SPI-stream alternate is parked as a **post-MVP optimization** to
revisit if:

- We add a board with no SBWO direction mux but with SPI pins routed to
  SBWDIO IN/OUT (unlikely for the current hardware roadmap), or
- The BSRR-script DMA-channel pressure becomes a problem on a future port
  (e.g. F0/L0 with tighter DMA budgets), or
- We want a single shared encoder template covering JTAG and SBW with the
  same code generator.

Filename `DtrigSbwSpiAlt.h` is reserved if/when implementation begins.

## Contingency: TIM-clocked asymmetric SBWCLK

If `t(SBW,Rtd)` validation shows the read slot needs more low-time than a
uniform SBWCLK can give at the target speed, generate **SBWCLK from a TIM
channel** (variable per-slot) and use SPI only to shift the SBWTDIO data.
SBWTCK is the target TEST/TCK pin — **not** loaded by the reset RC — so the
clock can stay crisp while the TDO slot's low-time is stretched to let the
target drive. This keeps the TMS/TDI slots fast and only widens the read slot,
recovering average throughput.

Cost: loses the "reuse DtrigJtag's SPI-generated clock verbatim" simplicity —
the SPI is now a slave/data-only shifter clocked alongside the TIM, and phase
alignment between TIM-SBWCLK and SPI-data needs its own bring-up. Treat as the
fallback only if the uniform stream's read slot proves tight on a real target.
